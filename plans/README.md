## Recommended order

1. **001** first — it restores the only no-toolchain verification baseline (the
   scope guards) and adds a PR CI that runs them. Everything else is safer once
   this is green.
2. **002, 003** — independently fix the dead release pipeline and the dead
   security scanning. No dependencies; can run in parallel with 001.
3. **004** — best after 001–003 so the docs can describe commands/workflows that
   actually work.
4. The code plans (**005–010**) and direction plans (**011–013**) can follow in
   any order subject to the dependency column.

## Findings considered and rejected (so they aren't re-audited)

- **Hardcoded Supabase keys + Google client IDs in `ios/project.yml`,
  `android/app/build.gradle.kts`, `core/EnvConfig.kt`** — these are **publishable**
  anon keys (`sb_publishable_` prefix) and OAuth client IDs, which are public-by-
  design for mobile clients (they ship in the binary; RLS is the real access
  control). Not a vulnerability. Only revisit if the team decides to externalize
  build config; rotation is the remedy if a key is ever misclassified.
- **Android `observeComments` `while(true)` Flow "resource leak"** (reported by an
  audit pass) — **false positive**. A `flow { while(true){ emit; delay } }` is
  fully cancellable; `delay` is a coroutine cancellation point, so collection
  cancellation stops it. The real (minor) issue there was the silent
  `getOrDefault(emptyList())` swallow — covered by plan 006.
- **iOS reset-password token passed via deep-link URL** — that is the standard
  Supabase email-recovery flow (the recovery link *is* a URL bearing the token);
  transient in-memory hold is normal. LOW; not planned.
- **iOS admin App Intents present in the registry** — by design: server-gated and
  enforced by `EndpointPolicyTests` / the scope guards. Not a finding.
- **iOS `SessionStore` bootstrap race / SwiftData multi-context writes** — flagged
  at MED confidence by an audit pass but speculative without runtime evidence; not
  turned into a plan. Revisit only with a reproduced symptom.

  SDK in the audit environment). Findings are from reading; every plan ends in a
  build/test verification the executor must actually run.
- `graphify-out/` (a committed 3.5 MB generated knowledge graph) was not audited
  for content; whether it belongs in version control is a separate question.
- Coverage was hotspot-weighted across both platforms' data/auth/feature layers;
  `:map`, `:calendar`, `FECityPicker`, and the Map screens received lighter
  attention than the Plan/Explore/Saved/EventDetail/Auth paths.
