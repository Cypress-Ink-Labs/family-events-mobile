## Findings considered and rejected (so they aren't re-audited)

- **Hardcoded Supabase keys + Google client IDs** in `ios/project.yml`,
  `android/app/build.gradle.kts`, `core/EnvConfig.kt` — **publishable** anon keys
  (`sb_publishable_` prefix) + OAuth client IDs, public-by-design for mobile
  clients (ship in the binary; RLS is the real control). Not a vulnerability.
- **Android `observeComments` `while(true)` Flow "leak"** — false positive; the
  flow is cancellable (`delay` is a cancellation point). The real issue (silent
  `getOrDefault(emptyList())`) was fixed by plan 006.
- **iOS reset-password token in deep-link URL** — standard Supabase recovery flow;
  transient in-memory hold is normal. LOW; not planned.
- **iOS admin App Intents in the registry** — by design: server-gated + enforced
  by `EndpointPolicyTests` / scope guards.
- **iOS `SessionStore` bootstrap race / SwiftData multi-context writes** — flagged
  MED but speculative without runtime evidence; not planned. Revisit with a repro.
